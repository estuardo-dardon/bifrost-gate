Guia HowTo de Tuneles VPN

Esta pagina resume los flujos recomendados para crear tuneles StrongSwan desde la API de Bifrost Gate con administracion centralizada de conexiones, secretos y certificados.

Ruta protegida con Basic Auth: solo usuarios autorizados de documentacion pueden acceder.

Mapa de Componentes

Componente Endpoint Uso
Conexiones /api/connections Define parametros del tunel (IKE, CHILD SAs, autenticacion, pools y rutas).
Secrets /api/secrets Guarda credenciales IKE PSK o EAP. Relacion por IDs de identidad.
Certificados /api/certificates Crea CA y certificados de usuario/servidor para autenticacion por pubkey.
Vinculo certificado-conexion /api/connections/{connection_name}/certificate Adjunta certificado a una conexion existente cuando necesites automatizar identidad local o CA remota.

Diagrama de Flujo Operativo

1
Definir Modelo

Selecciona tipo de tunel: PSK, EAP, certificados o mixto.

2
Preparar Material

Crear secrets y/o certificados segun el modelo elegido.

3
Crear Conexion

Enviar POST /api/connections con config compatible StrongSwan.

4
Validar y Operar

Verificar carga, estado y aplicar rotaciones sin romper IDs.

Escenario A: Peer to Peer con PSK

Requiere: secret tipo ike con IDs que coincidan con local.id y remote.id.

Paso 1: Crear secret IKE

{
    "name": "peer-to-peer-psk",
    "secret_type": "ike",
    "config": {
        "secret": "SuperSecretPSK123!",
        "ids": ["1.1.1.1", "2.2.2.2"]
    }
}

Paso 2: Crear conexion

{
    "name": "peer-to-peer",
    "config": {
        "version": 2,
        "local_addrs": "%any",
        "remote_addrs": "2.2.2.2",
        "proposals": "aes256-sha256-modp1024!",
        "local": { "id": "1.1.1.1", "auth": "psk" },
        "remote": { "id": "2.2.2.2", "auth": "psk" },
        "children": {
            "net-net": {
                "local_ts": "10.0.0.1/32",
                "remote_ts": "10.1.1.1/32",
                "esp_proposals": "aes256-sha256!",
                "start_action": "start"
            }
        }
    }
}

Escenario B: Peer to Any con EAP

Requiere: un secret tipo eap por usuario remoto.

Paso 1: Crear usuario EAP

{
    "name": "rw-alice",
    "secret_type": "eap",
    "config": {
        "secret": "AlicePassword#2026",
        "id": "alice"
    }
}

Paso 2: Crear conexion

{
    "name": "peer-to-any-eap",
    "config": {
        "version": 2,
        "local_addrs": "%any",
        "local": {
            "id": "1.1.1.1",
            "auth": "pubkey",
            "certs": ["serverCert.pem"],
            "send_cert": "always"
        },
        "remote": {
            "id": "%any",
            "auth": "eap-mschapv2",
            "eap_id": "%any"
        },
        "children": {
            "net-any": {
                "local_ts": ["10.1.1.0/24", "10.2.1.0/24"],
                "start_action": "trap"
            }
        },
        "pools": "vpn-pool"
    }
}

Escenario C: Peer to Any con multiples usuarios por certificado

Requiere: CA + certificados de usuario firmados por esa CA.

Paso 1: Crear CA

{
    "name": "corp-ca",
    "common_name": "Corp VPN Root CA",
    "organization": "Bifrost Corp",
    "country": "GT",
    "days": 3650,
    "key_size": 4096
}

Paso 2: Crear certificados de usuario

{
    "name": "alice-cert",
    "ca_name": "corp-ca",
    "identity": "alice",
    "san": ["alice", "alice@corp.example"],
    "days": 825,
    "key_size": 4096
}
{
    "name": "bob-cert",
    "ca_name": "corp-ca",
    "identity": "bob",
    "san": ["bob", "bob@corp.example"],
    "days": 825,
    "key_size": 4096
}

Paso 3: Crear conexion

{
    "name": "rw-multi-user-cert",
    "config": {
        "version": 2,
        "local_addrs": "%any",
        "local": {
            "id": "vpn.example.com",
            "auth": "pubkey",
            "certs": ["vpn-gateway.pem"],
            "send_cert": "always"
        },
        "remote": {
            "id": "%any",
            "auth": "pubkey",
            "cacerts": ["corp-ca.pem"]
        },
        "children": {
            "rw-cert-users": {
                "local_ts": ["10.10.0.0/16", "10.20.0.0/16"],
                "start_action": "trap"
            }
        },
        "pools": "rw-pool"
    }
}

Escenario D: Certificado + usuario/clave

Autenticacion combinada: certificado del cliente + EAP para segundo factor.

Paso 1: Secret EAP por usuario

{
    "name": "alice-eap-2fa",
    "secret_type": "eap",
    "config": {
        "secret": "Alice2FA-Password#2026",
        "id": "alice"
    }
}

Paso 2: Conexion con auth2

{
    "name": "rw-cert-plus-password",
    "config": {
        "version": 2,
        "local_addrs": "%any",
        "local": {
            "id": "vpn.example.com",
            "auth": "pubkey",
            "certs": ["vpn-gateway.pem"],
            "send_cert": "always"
        },
        "remote": {
            "id": "%any",
            "auth": "pubkey",
            "auth2": "eap-mschapv2",
            "eap_id": "%any",
            "cacerts": ["corp-ca.pem"]
        },
        "children": {
            "rw-cert-eap": {
                "local_ts": "10.50.0.0/16",
                "start_action": "trap"
            }
        },
        "pools": "rw-pool"
    }
}

Checklist de Validacion

OK: IDs de /api/secrets coinciden con local.id y remote.id (PSK) o con eap_id/identidad de usuario (EAP).

OK: Certificados de cliente estan firmados por la CA que declaraste en remote.cacerts.

ATENCION: Al rotar secretos o certificados, manten identidad estable para no romper asociaciones.
